# Java Deprecation Runbook — retiring `/java` (`com.tictactec`) for a generated, Maven-published artifact

**Status:** 📝 **PLANNED — decisions locked, implementation NOT STARTED (2026-07-08).** Coordinates,
abstract-layer fate, build tool, website posture, and removal gating are all decided (see §6). No
Java-build code has changed yet. The only thing done so far is a docs cosmetic: the website's
per-function Java link now points at `ta_codegen/output/java/Core_<NAME>.java` instead of the
un-renderable 3.2 MB `java/.../Core.java` monolith.

> Like `docs/canonical-cutover-runbook.md`, this is a *careful, reversible, one-step-at-a-time*
> plan. Every phase has a **green/red checkpoint** and a **rollback**. Do not proceed past a red
> checkpoint. Coordinate each phase with the maintainer. The guiding constraint: **existing
> `com.tictactec:ta-lib` users must never be stranded** — dual-publish and a relocation POM before
> anything is removed.

---

## 1. Where we are today

Two Java trees exist, and they are *not* the same thing:

| Tree | What it is | Generated by ta_codegen? |
|---|---|---|
| `java/` (legacy, shipped) | Full buildable library: Ant `build.xml`, checked-in `ta-lib.jar`, package `com.tictactec.ta.lib`, the 3.2 MB monolithic `Core.java`, `CoreAnnotated.java`, ~43 top-level source files, a 26-file `meta/` + `meta/annotation` reflection layer, and JUnit tests. | **Partially.** Only `Core.java` (GENCODE section), `CoreAnnotated.java`, and `FuncUnstId.java` are generated in place. Everything else is hand-maintained legacy. |
| `ta_codegen/output/java/` | 161 `Core_<NAME>.java` **method-body fragments** (no `package`, no `class`, no imports — they are spliced into the monolith) + `TaCodegenServe.java` (the JSON-RPC test server). | Yes, but it is **not a library** — no package decls, no assembled class, no shared types, no build file. |

So "output/java is the new way" is a *destination*, not the current reality. Closing the gap is the
work below.

**Naming reality check (the hyphen):** three different naming systems, three different rules —
- domain (`ta-lib.org`, `ta-lib.github.io`): hyphen **allowed**
- Maven groupId (`io.github.ta-lib`): hyphen **allowed** (it's a coordinate string)
- Java package (`package io.github.talib;` + folder path): hyphen **forbidden** (it's the minus operator)

They may differ; the only hard rule is "no hyphen in the Java package."

## 2. Decision A — Maven coordinates (chosen: GitHub-anchored)

**Chosen:**
```
groupId:    io.github.ta-lib      # verified via the GitHub org TA-Lib, NOT via a domain
artifactId: ta-lib               # the brand lives here (hyphen fine in artifactId)
package:    io.github.talib       # hyphen dropped (Java rule); imports read io.github.talib.*
coordinate: io.github.ta-lib:ta-lib:<version>
```

**Why GitHub-anchored, not domain-anchored:** the maintainer may not renew `ta-lib.org` after
retirement. A domain-based namespace (`org.ta-lib`) is verified by DNS and would become claimable by
whoever buys the lapsed domain. `io.github.ta-lib` is verified by proving control of the GitHub org —
no DNS, no renewal, survives as long as the org exists. Bonus: `io.github.ta-lib` is the reverse-DNS
of `ta-lib.github.io`, so a future GitHub Pages site and the Maven namespace are the *same identity*.

**Migration for existing users:** `com.tictactec:ta-lib:0.4.0` **is** on Maven Central (see §7). It is
immutable and stays resolvable forever, so nothing we publish can break existing pins. A **relocation
POM** to auto-forward old users is *optional* and depends on still holding Sonatype rights to the
`com.tictactec` namespace — details and fallback in §7.

**Website / github.io note (why the GitHub anchor is safe):** the site stays at **ta-lib.org** — it is
*hosted on* GitHub Pages (the `mkdocs gh-deploy` job pushes to the `gh-pages` branch of
`TA-Lib/ta-lib`; a `CNAME` file pins the custom domain `ta-lib.org`). Because the content lives on
GitHub, it is *also* reachable at the project Pages URL **`https://ta-lib.github.io/ta-lib/`**, which
today **301-redirects to ta-lib.org** (verified 2026-07-08 — normal behaviour when a custom domain is
set). If `ta-lib.org` is ever dropped, removing the `CNAME` makes `ta-lib.github.io/ta-lib` serve the
site directly. So GitHub is the durable home; the domain is a convenience layer on top. The Maven
namespace `io.github.ta-lib` is verified against the **GitHub org** (a repo Sonatype names), *not* the
Pages URL or the domain — so it is unaffected by dropping ta-lib.org. (A clean, path-less
`ta-lib.github.io` would require an org Pages repo named `TA-Lib.github.io`; not needed, and irrelevant
to Maven.)

## 2b. Decision C — build tool (chosen: Maven)

The "build tool" is the program that compiles the `.java` sources, packages them into a `.jar`, and
publishes to Maven Central. Today `/java` uses **Ant** (`build.xml`) — dated, imperative, no
dependency/publish story. The two modern choices are **Maven** and **Gradle**.

**Chosen: Maven.** For a *generated, single-module* library it is the right modern default:
- **Declarative & trivial to emit** — one `pom.xml` the generator writes; no scripting.
- **It *is* the coordinate system** — the artifact's `groupId:artifactId:version` are literally Maven
  coordinates; publishing to Central is Maven's native path (sources jar, javadoc jar, GPG signing).
- **Lowest friction for consumers** — the canonical `<dependency>` snippet everyone copy-pastes.

Gradle's advantages (incremental builds, flexible DSL, multi-module graphs) don't apply to a single
generated artifact and cost complexity. Pick Maven.

## 3. Decision B — `ta-abstract` / the Java introspection layer

This is the subtle one, because **"ta_abstract" means two different things** and only one of them is
in scope for deprecation.

### 3a. C `ta_abstract` — **KEEP. Not deprecated. Not affected.**

`src/ta_abstract/` (generated by `ta_codegen`'s `ta_abstract_c.rs`) is the canonical introspection
layer: `TA_GetFuncHandle`, `TA_FuncInfo`, `TA_ParamHolder`, `TA_CallFunc`, `TA_ForEachFunc`, the group
index, the func API tables. It is **load-bearing and widely used**:
- it drives `TA_ForEachFunc` enumeration in `ta_regtest` and the generic `test_codegen.c` callback;
- the JSON-RPC servers are built on it;
- **it is the foundation of the Python wrapper's `talib.abstract` API**, which real users depend on.

Nothing here changes. This runbook does **not** touch C `ta_abstract`.

### 3b. The **Java** abstract layer — `CoreAnnotated.java` + `meta/` + `meta/annotation/` — **decide.**

This is a *Java-native* reimplementation of the same introspection idea: `@FuncInfo`-style
annotations on `CoreAnnotated`, plus a `meta/` reflection framework (`TaFuncService`,
`CoreMetaData`, `TaFuncMetaInfo`, `PriceHolder`, …) that lets you enumerate functions, discover
their params, and invoke them dynamically by metadata. In this repo its **only consumers are its own
test files** (`test/TestAbstractClosure.java`, `test/TestCoreMetaData.java`).

**Why modern Java users rarely touch it (the claim, substantiated):**
1. **Direct calls are simpler and type-safe.** The common path is
   `new Core().rsi(startIdx, endIdx, in, period, outBeg, outN, out)` — a normal typed method with
   compile-time checking. The abstract layer only earns its keep when the function is chosen *at
   runtime from a string* (charting GUIs, backtesters with user-selectable indicators, param-sweep
   tooling). Most apps hardcode the handful of indicators they need.
2. **It fights the type system.** The reflection/annotation path uses `Object[]` param holders,
   boxing, and runtime dispatch — slower and clumsier than a direct call, and it throws away the
   compile-time safety that is the main reason to use Java.
3. **The dynamic-discovery niche moved elsewhere.** Users who genuinely want runtime introspection
   overwhelmingly reach for the Python `talib.abstract` wrapper (built on C `ta_abstract`), not a
   pure-Java reflection framework. The pure-Java dynamic-dispatch audience is small.
4. **Maintenance cost is high per user.** ~26 hand-maintained annotation classes + a generated
   `CoreAnnotated` that must stay in lockstep with every function signature change — a large surface
   for a feature few call.

**Options for the new artifact:**

| Option | What ships | Cost | When it's right |
|---|---|---|---|
| **B1 — Drop (recommended default)** | `Core` + enums + `MInteger` only. No annotations, no `meta/`. | Lowest. Matches how ~everyone uses the library. | No known Java consumer needs runtime function discovery. |
| **B2 — Regenerate from ta_codegen** | Generator emits `CoreAnnotated` + the annotation classes so they stay in lockstep for free. | High (teach the generator the whole annotation surface). | A real Java user needs dynamic dispatch and won't move to another route. |
| **B3 — Lightweight static metadata** | Generated, reflection-free metadata table (names, params, ranges) as plain data — introspection without the annotation/reflection framework. | Medium. | We want *discoverability* (list functions/params) but not dynamic invocation. |

**DECIDED (2026-07-08): B1 — drop the Java abstract layer entirely.** The new artifact ships `Core` +
enums + `MInteger` only. No `CoreAnnotated`, no `meta/`, no `meta/annotation/`. Java users call the
typed methods directly; anyone needing runtime introspection uses C `ta_abstract` (via the Python
`talib.abstract` wrapper or their own binding), which is unaffected.

## 4. The gap to close (Phase 1 scope)

To make `output/java` a self-contained, buildable, publishable library, `ta_codegen` must generate:
- **package declarations + an assembled `Core` class** (today the fragments have neither);
- **the shared types** — `RetCode`, `MAType`, `Compatibility`, `CandleSetting`/`CandleSettingType`,
  `MInteger`, `RangeType`, `MoneyFlow` — as templates under `ta_codegen/generator/templates/java/`, mirroring
  how C and Rust shared types already live there;
- **`FuncUnstId`** (already generated) relocated into the new package;
- a **`pom.xml`** (Decision C: Maven) with the chosen coordinates and Central-publish metadata
  (sources jar, javadoc jar, GPG signing);
- **nothing for the abstract layer** (Decision B1: dropped) — no `CoreAnnotated`, no `meta/`;
- a minimal or zero in-tree test surface — cross-language correctness is already covered by
  `ta_regtest` driving `TaCodegenServe`, so in-tree JUnit can likely be dropped.

## 5. Phased plan (each phase: checkpoint + rollback)

### Phase 0 — Docs + intent (in progress)
- ✅ Website Java links point at per-function `ta_codegen/output/java/Core_<NAME>.java`.
- Announce the deprecation direction (this runbook; a changelog/README note).
- **Checkpoint:** docs build clean, links resolve on GitHub. **Rollback:** revert the link change.

### Phase 1 — Make `output/java` a complete buildable Maven project
- Add Java shared-type templates under `ta_codegen/generator/templates/java/`.
- Teach `ta_codegen` to emit package decls, an assembled `Core`, the shared types, `FuncUnstId`, and
  a Maven `pom.xml`, all under the new package `io.github.talib`. No abstract layer (Decision B1).
- **Checkpoint (RED until all green):** `mvn -q compile` succeeds; `TaCodegenServe` still builds and
  `ta_regtest --codegen --language=java` is **161/0**; regen oracle shows only expected files moved;
  C library + `include/` ABI untouched. **Rollback:** the new tree is additive — delete it; `/java`
  is still the shipped path, nothing lost.

### Phase 2 — Dual-publish one release
- Publish `io.github.ta-lib:ta-lib:<version>` to Maven Central **and** keep `/java` building/publishing
  under `com.tictactec` for the same release.
- **Checkpoint:** both artifacts resolve from a clean consumer project; a smoke test calls `rsi`/`sma`
  on each. **Rollback:** unpublish/stage only — never release a broken new artifact.

### Phase 3 — Deprecate `/java`
- README + changelog notice; `@Deprecated` on the legacy entry points if kept a release longer.
- Relocation POM on `com.tictactec:ta-lib` → `io.github.ta-lib:ta-lib`.
- Repoint any remaining docs/build references from `java/` to `ta_codegen/output/java/`.
- **Checkpoint:** old-coordinate build resolves via relocation with a warning, not an error.
  **Rollback:** remove the relocation POM / deprecation notices.

### Phase 4 — Remove `/java`
- **Gating (maintainer's call, 2026-07-08): remove `/java` as soon as the rebuild from
  `ta_codegen/output/java` is complete and verified** — not tied to a fixed time window. Once
  `output/java` is a complete, compiling, regtest-green replacement (Phase 1 green), delete the
  `java/` tree (Ant build, checked-in jar, `com.tictactec` sources, meta layer) and its build-system
  references. (Phases 2–3 concern the *published Maven artifact* migration and can proceed in
  parallel; they do not block removing the in-repo `java/` folder.)
- Update `scripts/build.py`, source-list checks, and CI (mirror the gen_code Stage-7 removal pattern).
- **Checkpoint:** full `regtest.py` green; `--codegen` 161/0 all languages; no dangling `java/`
  references in CMake/autotools/CI/docs. **Rollback:** `git revert` the removal commit; the artifact
  is already independently published, so consumers are unaffected regardless.

## 7. Old artifact on Maven Central — RESOLVED (2026-07-08)

**Yes, it was published: `com.tictactec:ta-lib:0.4.0` is live on Maven Central** (confirmed via the
Central search API). Consequences:

- **Nobody is broken by anything we do.** Central is immutable — `0.4.0` stays resolvable forever for
  anyone with it pinned. The new library is a separate coordinate; publishing it can't disturb `0.4.0`.
- **The new artifact is a fresh listing**, `io.github.ta-lib:ta-lib`, versioned to match the library
  line (e.g. `0.7.x`), *not* a continuation of the `com.tictactec` `0.4.x` series.
- **Relocation POM is optional and has a prerequisite.** A `<relocation>` POM would auto-forward old
  users, but it must be published as a *new version under the old coordinate* (e.g.
  `com.tictactec:ta-lib:0.4.1`), which requires still holding **Sonatype publish rights to the
  `com.tictactec` namespace** (originally verified via the `tictactec.com` domain, years ago).
  - If that access still exists → publish a relocation POM in Phase 3 (cleanest).
  - If it's lost/unrecoverable → skip it. `0.4.0` remains available and untouched; announce the move
    via README/docs/changelog and the ta-lib.org site. No user breaks either way.

**Follow-up for the maintainer:** do you still control the Sonatype/`com.tictactec` publishing
account? Only needed to decide relocation-POM vs. docs-only announcement — not a blocker for anything.
