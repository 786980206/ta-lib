# Rust Codegen Output Restructure & Server Generation

## Problem

The ta_codegen pipeline generates JSON-RPC servers for C, Java, and .NET — all 163 indicators pass regression tests. Rust is skipped. The Rust backend generates indicator `.rs` files flat into `ta_codegen/output/rust/`, but there's no Cargo crate structure, no module declarations, and no server harness.

The old `rust/` top-level directory is a legacy Cargo crate from the `gen_code.c` era. It should be replaced by a fully generated crate in `ta_codegen/output/rust/`.

## Goal

1. Restructure `generate --backend=rust` to emit a complete, self-contained Cargo crate in `ta_codegen/output/rust/`
2. Generate a Rust JSON-RPC server binary within that crate
3. Pass all 163 indicators in `ta_regtest --codegen --language=rust`

## Reference

- C server: `server_gen.rs::generate_c_server` — server pattern reference
- Generated Rust indicators: `ta_codegen/output/rust/*.rs` — existing output (currently flat)
- Old Rust crate: `rust/src/ta_func/mod.rs` — Core, RetCode, FuncUnstId, Compatibility, TaFloat definitions to migrate
- Old Rust crate: `rust/src/ta_func/float.rs` — TaFloat sealed trait implementation

## Part 1: Crate Structure Restructure

### Current state

`generate --backend=rust` outputs flat `.rs` files to `ta_codegen/output/rust/`:
```
ta_codegen/output/rust/
  sma.rs
  rsi.rs
  bbands.rs
  ... (163 files)
```

No `Cargo.toml`, no `mod.rs`, no module declarations. These files can't compile.

### Target state

`generate --backend=rust` outputs a complete Cargo crate:
```
ta_codegen/output/rust/
  Cargo.toml
  src/
    lib.rs
    ta_func/
      mod.rs          (Core, RetCode, FuncUnstId, Compatibility, mod declarations)
      float.rs        (TaFloat sealed trait)
      sma.rs
      rsi.rs
      bbands.rs
      ... (163 indicator files)
    bin/
      ta_codegen_serve.rs   (JSON-RPC server — generated by generate-servers)
```

### Changes to `generate --backend=rust`

**Responsibility: `rust_lang.rs` backend + `main.rs` orchestration**

1. **Output path:** Change indicator file output from `ta_codegen/output/rust/<name>.rs` to `ta_codegen/output/rust/src/ta_func/<name>.rs`

2. **Emit `Cargo.toml`:** Generated with:
   - `name = "ta-lib"`, matching the old crate
   - `serde_json = "1"` dependency
   - `[[bin]]` target for `ta_codegen_serve`

3. **Emit `src/lib.rs`:**
   ```rust
   #![allow(non_snake_case, unused_variables)]
   pub mod ta_func;
   pub use ta_func::*;
   ```

4. **Emit `src/ta_func/mod.rs`:** Contains:
   - `RetCode` enum with derive traits
   - `Compatibility` enum
   - `FuncUnstId` enum (all 24 + `FuncUnstAll`)
   - `mod float; pub use float::TaFloat;`
   - `Core` struct with `unstable_period`, `compatibility`, `candle_settings` fields
   - `Core::new()`, `set_unstable_period()`, `set_compatibility()`
   - `mod sma;`, `mod rsi;`, etc. for all 163 indicators

5. **Emit `src/ta_func/float.rs`:** The `TaFloat` sealed trait. Embed the content as a const string in `rust_lang.rs` (copied from the current `rust/src/ta_func/float.rs`).

### Why the indicator backend, not server gen

The Cargo scaffolding (`Cargo.toml`, `lib.rs`, `mod.rs`, `float.rs`) is needed for the indicator code to compile. It's the indicator backend's responsibility to produce a working library crate. The server gen just adds a binary target into that crate.

## Part 2: Server Generation

### `generate-servers --backend=rust`

**Responsibility: `server_gen.rs`**

Emits `ta_codegen/output/rust/src/bin/ta_codegen_serve.rs` — a JSON-RPC server binary. The crate structure from Part 1 must already exist.

### Server architecture

Same as C/Java/.NET servers:
- Read JSON from stdin line-by-line
- Parse method name, dispatch to indicator
- Call `core.<func>::<f64>(...)` with turbofish
- Serialize response to stdout with timing

### Protocol

Identical to other servers — no changes:
- Request: `{"method":"TA_SMA","params":{...}}`
- Response: `{"retCode":0,"outBegIdx":N,"outNBElement":N,"outReal":[...],"timing_ns":N}`
- Special: `list_functions`, `set_unstable_period`, `set_compatibility`

### Generic dispatch

`f64` only, matching other servers. Call with explicit turbofish: `core.sma::<f64>(...)`.

### Input naming

Uses `expand_input_names()` — same as other servers:
- Single real: `"inReal"`
- Multi real: `"inReal0"`, `"inReal1"`
- Price-expanded: `"inHigh"`, `"inLow"`, `"inClose"`

### RetCode mapping

```
Success → 0, BadParam → 2, AllocErr → 3,
OutOfRangeStartIndex → 12, OutOfRangeEndIndex → 13, InternalError → 5000
```

### Special method details

- `set_unstable_period`: Map `id` (i64) to array index. Reject `id >= FuncUnstId::FuncUnstAll as usize`. Cast `period` to `i32`.
- `set_compatibility`: Map `0 → Compatibility::Default`, `1 → Compatibility::Metastock`.

## Risk

**Medium.** The crate restructure is straightforward (output path change + scaffolding generation). The server generator is already implemented. The main risk is that the generated Rust indicator code may have compilation errors — the Rust backend (`rust_lang.rs`) may need fixes for patterns it doesn't handle correctly yet. Those would be addressed as compilation errors surface.
